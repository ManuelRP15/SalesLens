### #1 — 401 from the background with the Lightning host
**Tags:** tooling-api, auth

**401 from the background with the Lightning host**: the `sid` cookie from `*.lightning.force.com` is not valid for the REST/Tooling API. Always call `*.my.salesforce.com`. Solved with `toApiHost()`.

### #6 — `CustomObjectTranslation` / `Translations` / `GlobalValueSetTranslation` are not Tooling API objects.
**Tags:** tooling-api, metadata-api  
**Files:** `metadata-api.ts`

**`CustomObjectTranslation` / `Translations` / `GlobalValueSetTranslation` are not Tooling API objects.** Verified against the full alphabetical Tooling API object list — they don't appear there. The only programmatic access is the Metadata API (`listMetadata` + `retrieve` + `checkRetrieveStatus`), which is why `metadata-api.ts` exists.

### #7 — `CustomObjectTranslation` doesn't support the `*` wildcard
**Tags:** metadata-api

**`CustomObjectTranslation` doesn't support the `*` wildcard** in `retrieve()` — the exact `fullName`s (`ObjectApiName-language`) must be enumerated via `listMetadata` first. `Translations` (the global file), by contrast, **does** support `members: ["*"]` and returns one file per configured language in a single retrieve.

### #8 — `CustomObjectTranslation` has no top-level `label`/`pluralLabel` field.
**Tags:** metadata-api

**`CustomObjectTranslation` has no top-level `label`/`pluralLabel` field.** The object name translation (singular/plural) lives inside `caseValues`, keyed by grammatical case (`caseType = "Nominative"`, `plural = true/false`).

### #9 — The retrieved zip only contains the *translated* value
**Tags:** tooling-api, picklist

**The retrieved zip only contains the *translated* value**, never the base one — except for `picklistValues` / `valueTranslation`, which conveniently include both `masterLabel` (base) and `translation` in the same element. Everything else (field/object/record-type/tab/app labels) needs a separate, tightly scoped Tooling API / REST Query API call for the base value.

### #10 — `FieldDefinition.DeveloperName` is unreliable
**Tags:** tooling-api

**`FieldDefinition.DeveloperName` is unreliable** (often blank even for genuine custom fields when queried via the Tooling API) — don't filter on it; scope by `EntityDefinition.QualifiedApiName` instead and simply look up only the keys you already know you need.

### #11 — `RecordType` is real CRM data, not a metadata-flavored object
**Tags:** tooling-api, record-type

**`RecordType` is real CRM data, not a metadata-flavored object** — querying it via `/tooling/query/` throws `INVALID_FIELD` on `DeveloperName` (the Tooling API's view of `RecordType` lacks that column). It must be queried through the regular REST Query API (`/services/data/vXX.0/query/`), where the real `RecordType` sObject does expose `DeveloperName`.

### #12 — fast-xml-parser's classic "collapses a single repeated element to an object instead of a 1-item array" bug
**Tags:** picklist, record-type, xml-parsing

**fast-xml-parser's classic "collapses a single repeated element to an object instead of a 1-item array" bug**: solved with a dedicated parser instance configured with `isArray: (tagName) => ARRAY_TAGS.has(tagName)` for every tag that's inherently repeatable in the translation file schemas (`fields`, `caseValues`, `picklistValues`, `recordTypes`, `customTabs`, `customApplications`, `valueTranslation`, etc.), instead of relying on ad-hoc array-wrapping after the fact.

### #13 — `atob`/`TextDecoder`-based zip handling works fine in an MV3 service worker
**Tags:** background

**`atob`/`TextDecoder`-based zip handling works fine in an MV3 service worker** (no DOM needed): base64 → `Uint8Array` via `atob` + a manual byte loop, decompression via `fflate.unzipSync`, and text decoding via `fflate.strFromU8` (UTF-8 by default), which is required since translation values routinely contain accented characters.

### #14 — `CustomFieldTranslation`/`RecordTypeTranslation` don't always use a flat `<label>` tag (caseValues fallback)
**Tags:** picklist, record-type  
**Files:** `metadata-api.ts`

**`CustomFieldTranslation` (and the object-level name, and `RecordTypeTranslation`) don't always use a flat `<label>` tag.** For languages with grammatical gender (verified against real org data for Dutch, and Spanish behaves the same way), Salesforce omits `<label>` entirely and instead nests a `<caseValues>` list (same shape as the object-level singular/plural mechanism: `plural` + `value`, optionally `caseType`). `parseObjectTranslationFile` in `metadata-api.ts` falls back to extracting from `caseValues` whenever `label` is absent, for fields, record types, and the object name alike (`extractCaseLabels` helper). Also: `<caseType>` is only present when a language has more than one grammatical case (e.g. German Genitive/Dative) — when it's absent, the value **is** the base/nominative form and must not be filtered out. Filtering on `caseType === "Nominative"` unconditionally (skipping everything when the tag is missing) was the actual root cause of fields/record types/picklist values silently never appearing, even though the base labels were being resolved correctly and the zip genuinely contained the data.

### #15 — `retrieve()` only includes a field's `CustomFieldTranslation` (and, presumably, a record type's `RecordTypeTra…
**Tags:** soap, metadata-api, picklist, record-type, xml-parsing

**`retrieve()` only includes a field's `CustomFieldTranslation` (and, presumably, a record type's `RecordTypeTranslation`) inside `CustomObjectTranslation` when the corresponding `CustomField`/`RecordType` component is ALSO present in the same retrieve package.** Verified two ways: (a) an independent retrieve via Workbench requesting only `CustomObjectTranslation` reproduced the exact same incomplete file our extension got (ruling out any bug in our SOAP client/parsing); (b) retrieving the same `CustomObjectTranslation` member together with the object's `CustomField` components made the missing `CustomFieldTranslation` entries (real labels and picklist values) appear immediately. Untranslated fields show up as `<label><!-- Master Label --></label>` — an XML **comment**, not real text — which fast-xml-parser correctly drops, so `field.label` is empty/falsy and no bogus entry gets created; this is expected, not a bug. Fix: `fetchMetadataTranslationEntries` now fetches `fieldLabels`/`recordTypeLabels` (via `FieldDefinition`/`RecordType`) **before** calling `retrieveMetadataZip`, and feeds their keys in as `CustomField`/`RecordType` members alongside `CustomObjectTranslation` in the same retrieve package.

### #16 — The `CustomObject`-sibling-unlock retrieve trick alone did NOT solve standard-field labels (see #21 instead)
**Tags:** metadata-api

**The `CustomObject`-sibling-unlock trick (adding `CustomObject: <object>` to the retrieve package, the same pattern as lesson #15's `CustomField` trick) was a reasonable, Salesforce-guidance-backed hypothesis for surfacing standard field/object label overrides from "Rename Tabs and Labels" — but it wasn't sufficient on its own in practice** (real-org testing: still "Unknown Origin" for standard fields even after modifying them via Rename Tabs and Labels). It's kept in the retrieve package since it's harmless and may still help for some override cases, but it is **not** the mechanism that actually solves standard-field translations — see lesson #21, which is.

### #17 — MV3 service workers get killed by Chrome after a short idle period (in practice, well under 2 minutes), and a…
**Tags:** resolution, background, testing

**MV3 service workers get killed by Chrome after a short idle period (in practice, well under 2 minutes), and a module-level variable is not a caching strategy.** The reverse index must be persisted to `chrome.storage.local` and restored on module load, or the extension silently reverts to whatever the module-level variable's initializer was (here, mock data) until something re-triggers a full reload — no amount of "it worked when I tested it a minute ago" testing will catch this without leaving the tab idle for a few minutes first.

### #18 — A hover tooltip that needs to be clickable can't rely on `relatedTarget` chains.
**Tags:** hover, tooltip-ui

**A hover tooltip that needs to be clickable can't rely on `relatedTarget` chains.** The pointer crosses arbitrary unrelated elements on its way from the hovered element to a tooltip rendered at a fixed offset, each firing its own mouseout; checking `event.relatedTarget` only catches the *last* hop. A grace-period timer (schedule a clear on `mouseout`, cancel it on the tooltip's own `mouseenter`) is robust regardless of what's in between.

---

### #19 — A floating, absolutely-positioned overlay that annotates dozens of elements across a real page will overlap an…
**Tags:** —

**A floating, absolutely-positioned overlay that annotates dozens of elements across a real page will overlap and won't read as "attached" to anything**, no matter how carefully positions are computed — real UIs are too dense and dynamic for that to hold up. Appending a small element as the *last child of the actual matched element* (real DOM insertion, not a separate shadow-root overlay) trades away the "we never touch Salesforce's DOM" guarantee for something that actually reads as attached to the right thing, with zero manual positioning code — see PHASE 9 below.

### #21 — Salesforce's own out-of-the-box standard translations for standard objects/fields/picklist values (the ones th…
**Tags:** soap, metadata-api, partner-api, picklist  
**Files:** `src/shared/describe-api.ts`

**Salesforce's own out-of-the-box standard translations for standard objects/fields/picklist values (the ones that "just exist" in every supported language, whether or not anyone has ever customized anything) are not exposed through `CustomObjectTranslation` at all — they come from the Partner API's `describeSObjects()` call with a `LocaleOptions` SOAP header** (`{ language: "fr" }`, etc.), which returns object `label`/`labelPlural`, every field's `label`, and every picklist value's `label`, all translated into whatever language `LocaleOptions` specifies — independent of the running user's own language, and independent of any Translation Workbench/Rename Tabs and Labels customization. This uses a different endpoint (`/services/Soap/u/{version}`, Partner WSDL namespace `urn:partner.soap.sforce.com` — see lesson #25, this was originally miswritten as `/c/` and didn't actually work until that fix) than the Metadata API (`/services/Soap/m/{version}`) — see `src/shared/describe-api.ts`. Practical consequence for the data model: `FieldLabel`/`ObjectLabel`/`PicklistValue` entries are now **seeded** from `describeSObjects` for every active language (plus the org's base language) covering literally every field/value describe knows about, and the `CustomObjectTranslation`-derived data (lesson #15) is layered on top as an override, tracked via `LabelEntry.customizedLanguages` so the UI can mark, per language, whether a given translation is Salesforce's stock one or an admin override.

### #24 — `resolveText`'s DOM-hint narrowing only checked one direction (`target.endsWith(candidate.apiName)`), so it si…
**Tags:** resolution, tooltip-ui, testing  
**Files:** `index-builder.ts`

**`resolveText`'s DOM-hint narrowing only checked one direction (`target.endsWith(candidate.apiName)`), so it silently failed to disambiguate whenever `data-target-selection-name` carried a bare field/object name shorter than the candidate's full `Object.Field` apiName** (e.g. hint `"MasterRecordId"` against candidate apiName `"Account.MasterRecordId"` — the hint is shorter than the apiName, so `endsWith` in that direction is always false). Real-org feedback (2026-07-19): a genuinely ambiguous case — the text "Account" is simultaneously a real Custom Label value, a real standard field label, and the real object label — stayed a 3-way tie in the tooltip even though the hovered element likely did carry a usable hint. Fixed in `index-builder.ts` by checking both directions (`target.endsWith(apiName) || apiName.endsWith(target)`). A second, weaker tie-break was added for cases where even that doesn't fully narrow it down (including when `targetSelectionName` is simply absent for that element — confirmed happening in real testing): candidates belonging to the object of the page currently being viewed (`hints.pageObjectApiName`) are sorted first — this **never claims `highConfidence: true`** off of this signal alone (per the "zero false positives" quality bar — see Product quality bars above), it only makes the ambiguous list more useful to read. **True 100% disambiguation from rendered text alone is not achievable when Salesforce genuinely uses the identical string for multiple unrelated metadata items in the same org** (as in this example) — closing that gap further requires richer DOM signals than `data-target-selection-name` alone provides, which is real, open work (see PHASE 8's detection heuristics).

### #25 — `describe-api.ts` was posting Partner-WSDL-namespaced SOAP envelopes (`urn:partner.soap.sforce.com`) to the EN…
**Tags:** soap, partner-api, editing  
**Files:** `describe-api.ts`, `metadata-api.ts`

**`describe-api.ts` was posting Partner-WSDL-namespaced SOAP envelopes (`urn:partner.soap.sforce.com`) to the ENTERPRISE API endpoint (`/services/Soap/c/{version}`) instead of the Partner API endpoint (`/services/Soap/u/{version}`) — so every single `describeSObjects` call faulted, for every language including the base language, from the day PHASE 7 shipped.** This is the actual root cause behind "only English and the admin-customized languages show a standard translation" (2026-07-19 real-org confirmation): `/c/` is the **Enterprise** API — a WSDL generated per-org with a namespace specific to that org's schema — while `/u/` is the **Partner** API — the generic, org-independent WSDL whose namespace is always `urn:partner.soap.sforce.com`, which is what our envelope was already correctly built for. Posting the right namespace to the wrong endpoint produced Salesforce's generic dispatch fault, `No operation available for request {urn:partner.soap.sforce.com}describeSObjects` — a fault that reads like a permissions/session problem but is purely a wrong-URL bug, confirmed by the fact that it faulted identically for every language (including en_US) and by the fact that `metadata-api.ts`'s SOAP calls against `/services/Soap/m/` — same session, same request shape — worked the entire time. The `[STI] describeSObjects coverage by language: ...` diagnostic line (added the same day, still in the code) was what made this checkable in one glance instead of guessing. **Lesson for future SOAP work in this project: the two-letter path segment after `/services/Soap/` (`c`, `u`, `m`, `T` for Tooling, etc.) determines which WSDL/namespace the endpoint expects — a namespace mismatch here reliably produces this exact "No operation available" fault, not an auth error, which is what makes it easy to mis-diagnose as an org/permissions issue instead of a URL bug.**

### #26 — Some ambiguity is genuinely resolvable from the DOM alone, via the tag name of well-known Salesforce base Ligh…
**Tags:** resolution, translation-health  
**Files:** `index-builder.ts`

### #29 — `findAttributeUpwards` in `dom-utils.ts` was skipping every same-tree ancestor
**Tags:** hover, resolution  
**Files:** `dom-utils.ts`

**`findAttributeUpwards` in `dom-utils.ts` was skipping every same-tree ancestor: it checked `getRootNode()` FIRST and, since in Lightning virtually every element lives inside some shadow tree, jumped straight from the hovered element to its shadow host — never visiting the element's own parents.** This is why `data-target-selection-name` came back `null` in real-org hover logs even for elements that plausibly had the attribute a few parents up in the same tree, quietly starving `resolveText` of its most reliable signal and forcing everything down to the weak tie-breaks. Fix: `parentAcrossShadow()` — `parentElement` always wins first; the shadow host is only the fallback once the current tree is exhausted (i.e. `parentElement` is null). `maxHops` also raised 6 → 20, since a correct walk now needs to cover Lightning's real nesting depth (the walk is cheap; the old low cap only existed because the broken walk ascended so fast). **Lesson: when an upward DOM walk must cross shadow boundaries, the boundary crossing is the LAST resort at the top of each tree, never the first move — and any "attribute is usually missing" symptom in Lightning deserves suspicion of the walk itself before concluding the attribute genuinely isn't there.**

**PHASE 6 inline editing (2026-07-20): scoped to Custom Labels only, by explicit product decision, because the two editable-metadata worlds in this codebase have fundamentally different write paths.** Custom Label translations are standalone Tooling API records (`ExternalString`/`ExternalStringLocalization`) that PATCH/POST individually, no different in kind from any other Tooling API write. Every other type this project resolves (`FieldLabel`, `ObjectLabel`, `PicklistValue`, `RecordType`, `CustomTab`, `WebLink`, `LayoutSection`, ...) lives **inside a `CustomObjectTranslation`/`Translations`/`GlobalValueSetTranslation` XML file** (lesson #6 — these aren't Tooling API objects at all), and the only way to change one value in that file is a full Metadata API `deploy()`: fetch the current file, patch one field, re-zip, base64, `deploy()`, poll `checkDeployStatus`, handle partial failures — a materially different, unbuilt pipeline, not a bigger version of the same PATCH call. Building both in one session risked shipping the second one half-finished (violates the project's no-half-finished-features stance); Custom Label editing is fully production-shaped, the rest is explicitly deferred to a future phase with its own deploy pipeline, gated behind `isEditableLabelType()` in `types.ts` so the tooltip's edit button simply doesn't appear for types it can't actually fulfill.
    **Base-language routing depends on a pre-existing assumption, now load-bearing:** `fetchAllTranslations` (lesson #5) has always seeded a Custom Label's org-default value under the literal key `"en_US"`, regardless of the org's *actual* default language — a simplification nobody had to think about while the key was read-only. Editing makes it load-bearing: `saveCustomLabelTranslation` checks `language === "en_US"` to decide PATCH `ExternalString.Value` (the base record) vs. PATCH/POST `ExternalStringLocalization` (an actual translation row) — the two are different objects with different semantics, and the base language never has (or needs) a localization row. **If a real org whose default language isn't English ever shows the base value being edited through the wrong path, both the seeding code and this routing check need to move together** — this hasn't been tested against such an org.
    **New vs. existing translation rows are distinguished by data already fetched, not by a follow-up query:** `ExternalStringLocalization`'s own `Id` is now captured per language into `LabelEntry.localizationIdsByLang` at load time (previously only `ExternalStringId`, the parent, was read — the row's own Id was never kept because nothing needed to address that specific row before). A language with an Id there PATCHes; a language without one POSTs a new row, and the returned Id is folded back into the in-memory entry so a *second* edit to a language in the same session PATCHes correctly instead of attempting a duplicate POST.
    **A focused, mid-edit `<textarea>` cannot be allowed to disappear out from under the user** — the tooltip already had three independent paths that could tear it down (mouseleave's grace-period clear, window `blur`, the popup's enable/disable toggle), any of which firing mid-keystroke would yank the input out of the DOM. Rather than guard each call site, a single module-level `isEditingActive` flag (set via a `Tooltip` → `CandidateBlock` → content-script callback whenever a row's editor opens/closes) is checked inside `clearTooltip()` itself in `content/index.tsx` — every teardown path already funnels through that one function, so one guard covers all of them by construction instead of by discipline.
    **Save/Cancel buttons need `onMouseDown={(e) => e.preventDefault()}`, a standard but easy-to-miss inline-editing gotcha**: clicking a button next to a focused `<textarea>` fires the textarea's `blur` (as focus moves toward the button) *before* the button's own `click` handler runs — without the guard, a blur-triggered auto-save/cancel would race the button's own action and the button's handler would sometimes act on already-torn-down state. Suppressing the mousedown's default focus-shift keeps focus on the textarea until the click handler explicitly decides what happens.
    **Missing-translation rows are shown for editable entries only, which extends rather than reverses the PHASE 4 product decision that rejected per-row "— not translated —" text** (see PHASE 4 additions above): that decision was about passive clutter — a warning nobody asked for, superseded by Translation Health. Once a row has an actual one-click action attached to it (add the missing translation right here), showing it stops being noise and starts being the entry point to the feature; non-editable types keep the exact old behavior (missing languages simply don't get a row).

### #42 — Optimistic concurrency control for edits (2026-07-20, product requirement, non-negotiable for production-readiness)
**Tags:** resolution, editing, concurrency, tooltip-ui  
**Files:** `salesforce-api.ts`, `Tooltip.tsx`

**Optimistic concurrency control for edits (2026-07-20, product requirement, non-negotiable for production-readiness): saving now ALWAYS re-reads the live org value first and compares it to the value the editor started from — never the local cache.** Root problem: the local reverse index can be stale (another user's edit, or this browser's own last refresh) for an arbitrary amount of time, so writing based on "what we last loaded" risked silently clobbering someone else's concurrent change with no trace. `fetchLiveCustomLabelValue` (`salesforce-api.ts`) does a `GET` on the exact record right before `saveCustomLabelTranslation`'s PATCH/POST — `ExternalString.Value` for the base language, `ExternalStringLocalization.Value` by Id for a known translation row, or a scoped SOQL query for a language with no known row yet (confirming one wasn't created concurrently, rather than assuming). A mismatch aborts the write entirely and returns `{ conflict: true, currentValue }` instead of writing — `SaveTranslationRequest.expectedValue` carries the editor's starting value (captured once in `CandidateBlock`'s `editingBaseline` state at the moment "Edit" is clicked, distinct from the live-typed value) all the way from `Tooltip.tsx` through the content script to the background. On conflict, the background folds the real current value back into its own cache (`entry.valuesByLang[language] = result.currentValue`) before responding, and the UI adopts it immediately (`setLocalValues`) with a transient banner explaining why the edit didn't take — "permitir refrescar antes de volver a editar" is satisfied for free since the row already shows the fresh value the moment the conflict is reported, no separate refresh action needed. Cost: one extra GET per save (an explicit, deliberate tradeoff — the correctness guarantee is worth the latency, per the requirement being "obligatorio para considerar production-ready", not optional).

### #43 — Hover system replaced end to end (2026-07-20)
**Tags:** translation-mode, hover, editing, tooltip-ui  
**Files:** `Tooltip.tsx`, `content/index.tsx`, `dom-utils.ts`

**Hover system replaced end to end (2026-07-20): hold-to-inspect → click-to-toggle Inspection Mode, and mouseover/mouseout → a single coordinate-driven engine — a redesign, not a patch series, per explicit direction ("no quiero un conjunto de parches").** Four real problems drove this, all fixed by the same underlying architecture change rather than four separate special cases:
    **(a) "The tooltip switches to a nearby element while I'm trying to reach it" (hover ownership).** Fixed geometrically, not by predicting mouse trajectory: `Tooltip.tsx` now reports its own settled `getBoundingClientRect()` up to the content script via a new `onRectChange` prop (fires after every clamp/flip AND after any resize, e.g. opening the inline editor makes the box taller). `content/index.tsx` keeps this as `tooltipRect` and checks it FIRST, before any DOM lookup, in `isWithinTooltipZone()` — a rect expanded by `TOOLTIP_OWNERSHIP_MARGIN_PX` (24px). While the cursor is inside that zone, `handlePointerMove` never retargets, full stop, regardless of what element is technically underneath along the way. A second, exact fallback (`rawEl === shadowHost`) catches the rare case where the rect is one frame stale.
    **(b) "Always Hover sometimes ignores elements, needs exit+re-entry" (detection reliability).** Root cause: native `mouseover`/`mouseout` events, which the old code relied on exclusively, go through Lightning's nested (and sometimes closed) shadow trees and don't fire reliably for every real transition there. Fix: **removed `mouseover`/`mouseout` entirely.** A single `mousemove` listener now drives everything through `handlePointerMove(x, y)`, which calls `deepElementFromPoint(x, y)` (now exported from `dom-utils.ts`) — recomputed straight from coordinates every time, sidestepping shadow-retargeting quirks completely instead of trusting the browser's own "did we enter a new element" computation. A cheap identity check (`rawEl === lastRawElement`) skips ALL further work (text extraction, attribute walks, messaging) when the element hasn't actually changed — this is what keeps a per-mousemove-event architecture cheap despite mousemove firing far more often than mouseover did: most movement lands on the same element as the last check, and those are now near-free.
    **(c) Hold-to-inspect → Inspection Mode (deliberate redesign, not a bug fix): press the configured key once to toggle inspection on — no holding.** The tooltip becomes a stable, fully interactive "pinned" surface: while the mode is on, `inspectAt` NEVER auto-clears on "nothing resolvable under the cursor" (only Always Hover still fades out after `CLEAR_GRACE_MS`) — moving to a genuinely different resolvable element still updates the tooltip (navigation by moving the mouse, the explicit ask), but empty/unresolvable space in between just leaves the last tooltip showing. Exit is multi-path by design: re-press the same key, Escape, clicking anywhere outside the tooltip, or losing window focus — all funnel through one `exitInspectionMode()`. `Settings.inspectorHotkey` keeps its exact existing meaning and storage key (`null` = Always Hover, unchanged) — only the keydown *behavior* changed from hold-while-down to toggle-on-press, so this shipped as a behavior refinement of an existing setting, not a new one.
    **(d) Eliminating stuck/inconsistent states — two real bugs, not hypothetical:** (i) `CandidateBlock`'s `isEditingActive`-reporting `useEffect` had no cleanup function — if the component ever unmounted while `editingLang` was still set (a path that didn't exist before this session but easily could in the future), the content script's `isEditingActive` flag would stay `true` forever, permanently blocking `clearTooltip()`. Fixed by returning `() => onEditingActiveChange?.(false)` from the effect — the standard React fix, and now the correct one to reach for the moment an "active edit" boolean is threaded across a component boundary like this. (ii) A NEW latent stuck-tooltip path this session's other changes could have introduced: `exitInspectionMode()`/`clearTooltip()` correctly no-op while `isEditingActive`, but nothing previously re-evaluated tooltip visibility once editing finished — if the extension got disabled, Translation Mode got toggled on, or Inspection Mode got exited (e.g. via window blur) WHILE an edit was in flight, the tooltip would linger forever with no further event ever able to clear it (the engine being correctly idle is exactly why nothing would trigger a clear on its own). Fixed with `reconcileAfterEdit()` — called once, exactly when `onEditingActiveChange` fires `false` — which just checks `isEngineLive()` and clears if it's no longer true. One general reconciliation step closes every one of these paths at once instead of special-casing each.
    **Design principle established for all future hover work: ownership and mode-exit guards belong in ONE shared function each (`isWithinTooltipZone`, `exitInspectionMode`), never duplicated at each call site** — every place that tried to be clever about "don't clear/retarget in situation X" before this session was exactly where a state bug had been found; centralizing the guard is what makes new call sites safe by construction instead of by remembering.

### #44 — Hover text detection now checks actual rendered glyph rects, not just the DOM box
**Tags:** hover, dom-heuristics
**Files:** `dom-utils.ts`

**Real-org complaint (2026-07-20): the tooltip resolved text "by the long side, where the cursor genuinely wasn't over it — the DOM just detected it was."** Root cause: `resolveHoverTarget` trusted `document.elementFromPoint(x, y)` to mean "the cursor is over this element's text," but `elementFromPoint` returns the innermost element at a pixel regardless of whether that pixel is real ink or just padding/line-height/block-level whitespace — a wide label, a full-width table cell, a padded container all have DOM boxes far larger than their visible text. Deliberately NOT applied to `collectTranslatableTargets` (Translation Mode's full-tree scan) — that mechanism has no cursor position at all, the fix is specific to point-based hover.

### #45 — Tooltip surface is pointer-events:none except its real controls, so it no longer blocks content it covers
**Tags:** hover, tooltip-ui
**Files:** `tooltip.css`, `content/index.tsx`

**Real-org complaint (2026-07-20): "when the tooltip lands on top of another scannable text, it doesn't get scanned — you have to click outside and re-click, or press Escape."** Root cause: `.sti-tooltip` was `pointer-events: auto` across its whole box (needed so its Copy/Edit buttons work), and lesson #43's tooltip-ownership check (`isWithinTooltipZone`) treated the ENTIRE rect+margin as permanently off-limits for retargeting, with no way to reach whatever the tooltip visually happened to be covering short of a hard reset. Accepted tradeoff: while moving the mouse toward a specific control, a different resolvable element directly behind the tooltip's background along that path could theoretically cause a brief retarget — judged a much smaller cost than the blocking behavior it replaces, and the same-text no-op check in `inspectAt` already absorbs most of it.

### #53 — PHASE 6b — Metadata deploy() pipeline, 8 more editable types
**Tags:** metadata-api, editing, concurrency, xml-parsing
**Files:** `metadata-api.ts`, `metadata-write.ts`, `metadata-write.test.ts`, `types.ts`, `background/index.ts`, `Tooltip.tsx`

**Product ask (2026-07-21): "complete PHASE 6 so all metadata can be edited as efficiently as possible."** Editing was scoped to Custom Labels only (`#41`) because every other type's translation lives inside a `CustomObjectTranslation`/`Translations`/`GlobalValueSetTranslation` XML file, only writable via a full Metadata API `deploy()` — a materially different, previously-unbuilt pipeline. This entry covers building that pipeline for 8 more types: `FieldLabel`, `RecordType`, `WebLink`, `QuickAction`, `LayoutSection`, `PicklistValue` (both field-scoped and global-value-set flavors), `CustomTab`, `CustomApplication`. `ObjectLabel`/`RelatedList` stay deferred and `StandardButton`/`StandardTab` stay permanently non-editable — see the scope section below.

**Why `deploy()` isn't as risky as it sounds:** it's transactional (`rollbackOnError=true`, `singlePackage=true`) — a malformed patch fails the WHOLE deploy with an error, it never partially applies. The worst case for a bug in this pipeline is "the save fails with a message," never silently corrupted org metadata. This is the reasoning that made building the pipeline an acceptable session-sized risk rather than something requiring a smaller/staged rollout.

**The single most important safety decision: preserveOrder XML, not the existing friendly-object parser, for writing.** `metadata-api.ts`'s existing `translationXmlParser` (friendly-object mode) is what makes reading pleasant — but it's lossy: fields it doesn't recognize, attribute ordering, comment nodes, all get dropped or reshaped on the way in, and there is no reliable way to serialize it back to valid XML that preserves everything the original file had (validation rules, sharing settings, other fields never touched by one edit). Rebuilding a whole file from a friendly-mode parse risked silently deleting content unrelated to the edit. Fix: a SEPARATE `preserveOrder: true` parser/builder pair (`parseXmlPreserveOrder`/`buildXmlPreserveOrder`, both new, `metadata-api.ts`) used ONLY by the write path — every node keeps its exact position/siblings/attributes, so "patch (or insert) exactly one node, leave everything else byte-for-byte" is actually safe. A handful of small AST helpers (`xmlFirstChild`/`xmlAllChildren`/`xmlLeafText`/`xmlSetLeafText`/`freshXmlDocument`) sit on top, all covered by `metadata-write.test.ts` against real fixture XML (patch-existing, insert-new-node, and build-fresh-file cases all verified to leave sibling content untouched) — this is the one part of this pipeline that's actually unit-tested end-to-end without a live org, precisely because it's the part where a subtle bug would be most dangerous.

**The second safety decision: lesson #15's sibling-unlock rule applies to writes exactly as much as it applied to reads, and getting it wrong risks real data loss, not just a wrong display.** `retrieve()` only includes a field's real `CustomFieldTranslation` (and record type's, web link's, quick action's, layout section's) content inside `CustomObjectTranslation` when the corresponding sibling component (`CustomField`/`RecordType`/`WebLink`/`QuickAction`/`Layout`, plus `CustomObject` for the object itself) is ALSO present in the SAME retrieve package — otherwise the read comes back as an XML comment placeholder, which the parser correctly drops as empty. If `metadata-write.ts` read the current value without the right siblings unlocked, it would see "" even when a real translation exists — and since the optimistic-concurrency check compares the read value against `expectedValue`, a stale/wrong "current value" of "" would make the concurrency check falsely PASS (real value silently overwritten) instead of correctly detecting drift. `metadata-write.ts`'s `resolveTarget()` builds the full sibling-unlock member list per type (mirroring the read path's own `metadata-translations.ts` construction) — including, for `LayoutSection` specifically, an extra `listMetadataFullNames(..., "Layout")` call, because a `LayoutSection` entry only knows its object + section name, not which literal `Layout` record it lives in (the read path flattens every layout's sections into one list) — so ALL of the object's layouts get unlocked rather than guessing which one.

**The third safety decision: never treat "couldn't retrieve" the same as "doesn't exist yet."** `retrieveMetadataZip` degrades gracefully on the READ path (returns `null` on any failure — network, auth, timeout, or a genuinely empty result, all collapsed into the same `null`). For a write, collapsing "retrieve failed" and "this file has never existed" into the same fallback ("build a fresh document from scratch") would be dangerous: deploying a fresh, near-empty file over an object that actually has many real existing translations — just because retrieve happened to fail this one time — would DESTROY those translations, since `deploy()` replaces the target component wholesale, it doesn't merge. `saveMetadataTranslation` THROWS when `retrieveMetadataZip` returns `null`, refusing to proceed at all; the "build fresh" path is only reached when retrieve() itself SUCCEEDED (a real zip came back) but the specific file's suffix genuinely isn't among the returned entries — the only evidence-backed signal available that it doesn't exist, not an assumption.

**Both write functions (`deployMetadataFile`/`checkDeployStatus` in `metadata-api.ts`, `saveMetadataTranslation` in `metadata-write.ts`) THROW on failure, unlike every read-side Metadata API helper in this codebase, which degrades to `null`/`[]`/empty `Map`.** This is a deliberate, DIFFERENT rule for writes vs. reads (see the updated rule #5/#6 in `CLAUDE.md`/`ARCHITECTURE.md`) — a save the user explicitly asked for must surface a real error, not vanish silently the way an unrelated failed read should. `background/index.ts`'s `saveTranslation()` already had exactly this catch-and-format-message shape for `saveCustomLabelTranslation` (`#42`); it now dispatches to either write function based on `labelType` and shares the same catch block, so no new error-handling shape was needed.

**Scope, and why each boundary is where it is:**
- **In scope, `CustomObjectTranslation` (per object+language):** `FieldLabel` (`fields[].label`), `RecordType` (`recordTypes[].label`), `WebLink` (`webLinks[].label`), `QuickAction` (`quickActions[].label`), `LayoutSection` (`layouts[].sections[].label` — patch-only, no insert: we don't know which `<layouts>` entry a never-before-translated section belongs to, and guessing wrong would silently misfile it), `PicklistValue` when field-scoped (`fields[].picklistValues[].translation`).
- **In scope, `Translations` (global, per language):** `CustomTab`, `CustomApplication`.
- **In scope, `GlobalValueSetTranslation` (per value-set+language):** `PicklistValue` when the picklist uses a global value set — disambiguated at write time exactly the way the read path already keys these entries: an `apiName` containing a `.` is field-scoped (`Object.Field#Value`), one without is a global value set (`GVSName#Value`).
- **Deferred, not this phase — `ObjectLabel`, `RelatedList`:** both ultimately target `<caseValues>` at the `CustomObjectTranslation` root (or, for fields/recordTypes, per-node), which can hold MULTIPLE grammatical-case entries for gendered languages (German, Slavic...). `assertNoGenderedCase` already guards `FieldLabel`/`RecordType` against this (refuses to patch a field/record-type whose language uses case forms, with a clear error, rather than silently writing to the wrong representation or corrupting the other cases) — the SAME complexity applies to the object's own label, just not yet built out into its own safe patch-or-refuse logic. `RelatedList`'s displayed value is entirely DERIVED from the related object's plural label (one hop removed from the same node), so it rides on whatever `ObjectLabel` editing eventually does, not its own separate mechanism.
- **Permanently non-editable, not a pipeline gap — `StandardButton`, `StandardTab`:** these come from Salesforce's own platform-controlled translations (`describeLayout` + `LocaleOptions`, or the bundled static catalog in `platform-labels.ts`) — there is no admin-authored value anywhere to write back to, ever. Worth being explicit that this is "by design," not "still TODO," so a future session doesn't go looking for a pipeline that can never exist.

**Insert vs. patch-only was an explicit product decision, not a default:** asked directly, given the real complexity/risk tradeoff (patch-only is simpler and lower-risk; insert also handles a translation that's never existed before, which is most of the real gap-filling work) — the answer was "both," so `locateOrCreateBlock` (shared by every type's target resolver except `LayoutSection`'s insert case, per above) always appends a well-formed new block rather than refusing when nothing matches yet.

**UI changes were almost nothing** — the whole point of building this behind `isEditableLabelType()`/`SAVE_TRANSLATION`'s existing shape: `Tooltip.tsx`'s editor, `translation-mode.tsx`'s clickable chips, and the missing-language "Add X translation" row all key off `isEditableLabelType()` already, so extending `EDITABLE_LABEL_TYPES` (`types.ts`) with the 8 new types lit them up automatically. The one deliberate addition: the Save button now shows "Deploying…" instead of "Saving…" for every type except `CustomLabel` (`TranslationEditor`'s new `savingLabel` prop, set in `CandidateBlock`) — this write is meaningfully slower (a retrieve + an async, polled `deploy()`, realistically several seconds to tens of seconds) than Custom Label's near-instant Tooling API PATCH, and the UI should set that expectation rather than look stuck.

**Real-org verification is mandatory and higher-stakes than usual for this one** — this is a genuine write to org metadata, not just a UI behavior. Recommend testing against a sandbox/dev org first, starting with one `FieldLabel` edit (both patching an existing translation and filling in a previously-missing one), before trusting it broadly.

### #54 — First real-org test of PHASE 6b: false-conflict blanking bug + tooltip made solid to persist on click
**Tags:** editing, concurrency, hover, tooltip-ui
**Files:** `metadata-write.ts`, `background/index.ts`, `tooltip.css`, `content/index.tsx`, `dom-utils.ts`, `metadata-write.test.ts`

First real-org run of PHASE 6b (2026-07-21) surfaced two problems the unit tests couldn't: non-Custom-Label editing "went very badly," and the hover tooltip still closed on any click inside it.

**Bug 1 (the serious one) — editing a metadata translation blanked the row and dropped the language ("desaparece la traducción y se quita el idioma directamente por la cara").** Root cause was in `saveMetadataTranslation`'s optimistic-concurrency check, which compared the wrong two things. `currentValue` is read from the deployable XML file — an admin OVERRIDE, or `""` when none exists. `expectedValue` is what the tooltip displayed — the EFFECTIVE label, which for these types is Salesforce's own STANDARD translation whenever the language was never customized (that standard value comes from `describeSObjects`/the platform catalog and **never lives in the CustomObjectTranslation file at all**). So for the overwhelmingly common case — editing a field/picklist/etc. whose shown value was a standard translation — `currentValue` (`""`) never equalled `expectedValue` (e.g. "Industria"), the check reported a bogus "someone else changed it" conflict, and the UI faithfully did what a real conflict demands: adopted the (empty) live value and dropped out of edit mode. The result looked like the translation vanishing on save. **Fix:** compare against the expected OVERRIDE, not the effective display value — extracted as the pure, unit-tested `isWriteConflict(currentValue, expectedValue, showedStandardValue)`. Two states are legitimately no-conflict: (a) the file already holds exactly what we displayed (a customized override we saw), or (b) the language showed a STANDARD value (`showedStandardValue`, i.e. not in `entry.customizedLanguages`) and the file still has no override — precisely the gap the edit fills. Anything else is genuine drift and still correctly conflicts, so the concurrency guarantee (rule #9) is intact, just no longer firing on phantom conflicts. **Companion fix in `background/index.ts`:** after a successful deploy()-backed save, the language is added to `entry.customizedLanguages` — it now HAS an override. Without this, a second edit of the same row in the same session would take the `showedStandardValue` branch again while the file already held our freshly-written override, reproducing the exact false conflict one step later (plus it makes the "✎ customized" mark appear immediately, which is now accurate). Custom Labels are left untouched — they have no standard/override split.

**QuickAction caveat found in the same session:** editing a global/standard action (New Contact, Log a Call…) failed with the raw metadata error "no QuickAction named Account.NewContact found". These actions are surfaced on an object's layout so they look object-scoped, but their translations don't live in that object's `CustomObjectTranslation` — and there is no real-org-verified route for them yet (global quick action translations would live in the global `Translations` file, but writing there is unconfirmed, and standard actions may be platform-controlled like standard buttons). Per this project's "don't guess" bar, `describeDeployFailure` now turns that specific rejection into a plain "This looks like a standard or global action — editing its translation from here isn't supported yet," instead of leaking the raw error or fabricating an unverified write. The deploy is transactional, so nothing was written and the row keeps its value. Object-specific custom quick actions are unaffected and still deploy normally. Full global-QA support is deferred (needs the seeding to carry a global flag and a real-org-verified `Translations` write path).

**Bug 2 — the tooltip still closed on a click anywhere inside it.** Lessons #45/#52 had made `.sti-tooltip` `pointer-events: none` (so clicks pass through to page content underneath) and tried to keep it open on inside-clicks via a coordinate/rect zone check (`isWithinTooltipZone`). That geometry could be a frame stale, and a pass-through click reported the underlying page element (not `shadowHost`) to the document listener, so some inside-clicks still read as "outside" and closed it. **Fix: make the tooltip a SOLID surface again (`pointer-events: auto`).** Because it lives in a CLOSED shadow root, every click that originates anywhere inside it retargets to `shadowHost` for the document-level listener, so a single `e.target === shadowHost` check keeps it open for ANY inside click — by construction, no geometry to go stale. Only a click on genuinely different page content, Escape, or the toggle key closes it now (exactly the behavior the user asked for: "debe persistir a no ser que clickes fuera o al escape"). Accepted tradeoff, reversing #45: you can no longer click *through* the tooltip onto the page content it visually covers — but in Inspection Mode the tooltip is offset from the cursor and re-anchors as you inspect new elements, so covered content is still reachable by moving/scrolling, and reliable persistence was the explicitly higher priority. **Responsiveness:** removed a `console.log` in `resolveHoverTarget` (`dom-utils.ts`) that ran with a `JSON.stringify` on essentially every pointer move — pure main-thread waste that made always-on hover feel laggy.

### #55 — Standard-field edits failed with "Element fields is duplicated"; Escape/outside-click were inert during an active edit
**Tags:** editing, xml-parsing, tooltip-ui
**Files:** `metadata-write.ts`, `metadata-write.test.ts`, `Tooltip.tsx`, `content/index.tsx`

**Bug 1 — editing a STANDARD field or picklist failed outright with "Error parsing file: Element fields is duplicated at this location in type CustomObjectTranslation"; custom fields/picklists worked fine.** Root cause: `locateOrCreateBlock`'s insert-new-node path (`rootChildren.push({[containerTag]: fresh})`) blindly appended the new block at the very END of the document's children array — after WHATEVER else was already there (`<validationRules>`, `<webLinks>`, `<layouts>`, ...). Salesforce's metadata XSD declares `CustomObjectTranslation`'s child elements in alphabetical order and requires repeated elements (like `<fields>`) to stay CONTIGUOUS; if the file already has a `<fields>` group followed by later-in-schema elements, and a brand-new `<fields>` block gets appended after those, the deploy validator sees TWO separate, non-contiguous `<fields>` runs and rejects the whole deploy as a duplicate. This is exactly why the symptom looked custom-vs-standard: a custom field with a PRIOR override already sits in the existing contiguous `<fields>` group, so editing it is a PATCH (no insert, no bug); a standard field being translated for the first time has no existing entry, so it's an INSERT — and virtually every standard field in a fresh/lightly-customized org falls into that bucket, while the test org's custom fields happened to already have prior overrides from earlier testing. **Fix:** a new `insertContainerNode` helper — if same-tag siblings already exist, insert right after the LAST one (keeps the group contiguous regardless of where it sits); otherwise insert right before the first sibling whose tag sorts alphabetically after the one being inserted, or at the end if none does. Regression-tested (`metadata-write.test.ts`) against the exact real-org shape: an existing `<fields>` group followed by `<validationRules>`/`<webLinks>`, inserting a never-before-translated field — the test fails against the old blind-push behavior and passes against the fix, confirmed by hand-tracing both.

**Bug 2 — Escape and clicking outside the tooltip did nothing while an edit was in progress; only the row's own Cancel button worked.** By design (lesson #43), `exitInspectionMode()`/`closeTmEditor()` both no-op while `isEditingActive`, to protect a mid-keystroke textarea from being torn out from under the user. The ORIGINAL plan relied on the keydown event bubbling past this capture-phase guard down to the focused `<textarea>`'s own local Escape handler — but that only works if the textarea still HAS focus, and it commonly doesn't: clicking the Save button disables it during the async save, which blurs it, and after a failed save (the standard-field bug above made this the common case) focus lands on `document.body`, not back on the textarea — so the local handler never fired either, leaving the user with no way out except Cancel, which real-org feedback flagged as unintuitive ("no se quita el modal... solo dandole a cancel"). **Fix:** a new `cancelTrigger` counter prop on `Tooltip`/`CandidateBlock` (same pattern as PHASE 17's `editTrigger`, just the opposite direction) lets `content/index.tsx` explicitly request "cancel whatever's being edited" — a new `requestCancelEdit()` re-renders the currently-shown tooltip (hover path: bump `cancelTrigger` and re-render via `showTooltip`; Translation Mode path: `renderTmTooltip` without `autoEditLanguage`, since that editor always force-remounts anyway per lesson #46, so a fresh mount with no auto-edit lands in view mode directly) with no dependency on DOM focus at all. Wired into both the Escape keydown branch and the click-outside handler, checked FIRST (before the existing `inspectionModeActive`/`tmEditorOpen` no-ops, which remain correct for when nothing is being edited). Deliberately scoped to match Cancel's own behavior — cancels the edit, tooltip stays open in view mode — not "also fully close the tooltip"; a second Escape/outside-click (now that editing has stopped) falls through to the normal exit/close path on its own, which reads as a natural two-step "back out" rather than an unexplained jump to fully closed.

Holding ALWAYS counts as "searching" even if something is already pinned (you're actively hunting for something new to replace it), which is why the formula can't simply gate on `currentTooltipText === null` alone.

**Conflict prevention, `shared/hotkeys.ts`:** the user explicitly asked that the two shortcuts can't be set to the same key, and to consider a small reusable mechanism rather than one-off logic — `normalizeBareKey`/`bareKeysConflict`/`pickAvailableBareKey` are it. `bareKeysConflict` powers two things: (1) `ShortcutToggleRow`'s `checkConflict` prop rejects a recorded key that collides with the OTHER shortcut's current key, showing an inline error message instead of silently accepting it (never silently overrides, per the user's explicit ask); (2) `pickAvailableBareKey` picks a safe key when a shortcut is RE-ENABLED (toggled from disabled back to on) — enabling always tries the setting's own default (Alt / Shift) first, only falling back through a small fixed pool (`["Alt","Shift","Control"]`) if that default happens to collide with the other shortcut's current custom key. Scope deliberately excludes `tmHotkey` (Toggle Translate all) — it's a combo (`"Alt+T"`), a fundamentally different trigger mechanism (a specific key while modifiers held, not the bare key itself firing), and conceptually different too: `null` there means "no shortcut assigned" for an already-independently-toggleable feature (`Settings.translationModeEnabled`), not "this feature is off." Extending conflict-checking to combos wasn't asked for and isn't the same kind of ambiguity.

### #60 — Translation Audit & Guided Navigation shipped ("Translate All" evolution) (2026-07-21)
**Tags:** translation-mode, editing, concurrency, product
**Files:** `content/translation-mode.tsx`, new `content/AuditPanel.tsx` + `content/audit-panel.css`, `content/index.tsx`, `docs/ROADMAP.md` (PHASE 18, full technical plan written before implementation, not duplicated here — read it for the seven architecture questions this entry assumes were already answered)

Directly-requested evolution of "Translate All" (Translation Mode, PHASE 9) from a display+edit mode into a guided audit workflow: filter to what needs attention, step through issues, watch the page scroll to and highlight each one, fix it with the existing editor, watch the count drop, repeat. Superseded PHASE 18's original, narrower scope ("Translation Navigator" read-only list + separate "Page Coverage" stats) — the richer version absorbs both (the list IS the Navigator; a per-filter count row is the Coverage stat) rather than building either separately.

**Everything reused, nothing duplicated — this is the load-bearing constraint the whole design serves:**
- **Scan/discovery:** `translation-mode.tsx`'s existing `scan()` (`collectTranslatableTargets` + one `RESOLVE_TEXTS_BULK` round trip) is the ONLY data source. It already walks the FULL page regardless of scroll position (`isRenderable()` checks `getBoundingClientRect().width/height > 0`, not viewport visibility — confirmed by reading `dom-utils.ts` directly rather than assuming) so below-the-fold issues are already in scope with zero new scanning logic.
- **DOM mapping:** `scan()` already produces `{element, entry}` pairs per matched text for badge placement — v1 just keeps that mapping around instead of discarding it after the badge is built, de-duplicated by `apiName + type` into a new exported `AuditEntry { key, entry, element, missingLanguages, identicalLanguages, editable, status }` list, handed out via a new `onAuditUpdate` callback parameter on `startTranslationMode()` (same "one scan, multiple consumers" shape the badges themselves already are, not a second scan).
- **Status computation:** missing/identical exactly reuse `#58`'s per-language checks and `#59`'s `BASE_LANGUAGE` constant — computed for EVERY resolved entry now, not gated by the badge's own `hasAnyValue` skip (that gate exists to avoid on-page annotation noise; the audit panel's entire purpose is to surface exactly the entries `hasAnyValue` would hide, e.g. an entry with zero coverage in any of the user's active languages — genuinely the worst case, not noise). One overall `status` per entry, priority `missing > identical > complete`.
- **Editor:** `content/index.tsx`'s existing `openTmEditor(entry, language, x, y)` — the exact function a Translation Mode chip click already calls — is the ONLY editor. Guided navigation computes `(x, y)` from the target element's own `getBoundingClientRect()` instead of a real click event, and passes the SPECIFIC language that made the entry match the filter (`missingLanguages[0]`/`identicalLanguages[0]`) as `autoEditLanguage`, not just the first configured language. Zero new editor code — concurrency control, the conflict banner, keyboard shortcuts all come along for free, same as every other consumer of this function.
- **Hover/Inspection Mode interaction:** untouched. `isEngineLive()` already returns `false` whenever `translationModeEnabled` is true, so the audit panel (which only exists while Translation Mode is on) can never contend with the hover tooltip for anything. No new mutual-exclusion logic was needed or added.

**New, because nothing existing covered it:**
- **`AuditPanel.tsx`** — a collapsed pill by default (`"N to review"`, matching PRODUCT.md's "never dump information" restraint), expanding into filter tabs (**All / Missing / Identical / Complete** — deliberately not the larger "Needs attention" union floated in the brief; each tab is a real, individually-computed status, adding a union tab would add a tab without adding information), a `"{Filter} — i of N"` stepper with Prev/Next, a clickable current-entry card, and a scrollable full list. Purely presentational — every interaction calls back into `content/index.tsx`, which owns all DOM/editor side effects.
- **Highlight overlay** (`highlightElement`/`positionHighlight`/`clearHighlight` in `content/index.tsx`) — a single floating `<div>` synced to the target's live rect on scroll/resize, with a brief CSS `@keyframes` pulse settling into a steady outline. Deliberately an OVERLAY, not a class/inline style applied to the target element itself — never risks fighting Salesforce's own CSS on the real element, and trivially, completely reversible (remove one node). Only one highlight ever exists — the exact "many simultaneous floating things overlapping" failure that killed Translation Mode's OWN v1 design (`#19`) doesn't recur here because guided navigation only ever highlights the ONE entry currently being worked on, never many at once.
- **A second, independent React root sharing the tooltip's shadow host.** `ensureHost()` (the tooltip) and the new `ensureAuditHost()` (the panel) now both call a shared `ensureShadowRoot()` and each create their OWN `createRoot()` mount point inside it, rather than each attaching a separate closed shadow root. Reasoning: CLAUDE.md rule #3 (closed Shadow DOM) is satisfied either way, but ONE shadow tree is simpler to reason about, and — the actual reason this matters, not just tidiness — two independent React roots means rendering one (e.g. `renderTmTooltip`'s `host.render(null); host.render(<Tooltip/>)`) can never blow away the other's mounted tree the way sharing a SINGLE root would have.
- **Save-triggered rescan (a real, previously-unnoticed gap, fixed as part of this work, not a pre-existing bug report):** an edit happens inside this extension's own CLOSED shadow root, so the page-level `MutationObserver` that normally re-triggers Translation Mode's rescan never fires from it. Before this entry, a saved translation edit would leave the on-page badge showing the STALE value until some unrelated page mutation happened to retrigger a scan — the audit panel would have inherited the exact same staleness. Fixed by calling `startTranslationMode(...)` again (an existing, already-safe "just rescan now" branch when already running) immediately after any `response.ok` save while Translation Mode is on, in `content/index.tsx`'s `saveTranslation()`. This is what makes "Missing: 12 → edit one → Missing: 11" (the brief's own worked example) actually happen instead of requiring a manual refresh.
- **Index-clamping instead of explicit "auto-advance":** `AuditPanel` re-clamps `currentIndex` against the FRESH filtered array on every render (`Math.min(currentIndex, filtered.length - 1)`). When a save removes the just-fixed entry from the filtered list, whatever now occupies that same numeric position becomes "current" for free — matching the brief's worked example without the panel yanking the page to a new position on its own initiative right after the user finishes typing (no scroll/highlight/editor-open fires until the user explicitly clicks Next/a row again).

**Duplicates — designed, deliberately deferred, not a gap left unconsidered.** The brief specifically warned against a misleading duplicate detector: "do not assume every repeated translation is a problem." The definition worked out (a per-active-language grouping by translated value, flagged ONLY when 2+ distinct on-page entries share a value AND their base-language values differ from each other — same value with the same base is EXPECTED, e.g. many honestly-different fields all legitimately named "Save" in English) is reliable and computable from data already fetched, but needs a real-org look at actual duplicate clusters before the panel can be trusted to surface them — same "don't guess at a heuristic without real bad-data examples" bar PHASE 10's QA Report v2 already follows. Full definition lives in `ROADMAP.md` PHASE 18, ready to implement as a small, well-specified follow-up (one more pass over already-collected `scan()` data, one more filter tab) rather than a vague TODO.

**Known limitations, real-org UNVERIFIED (same caveat as every round in this codebase, but this feature needs it more than most):**
- This is the most interaction-heavy feature shipped yet — scroll timing, highlight positioning across Lightning's nested/shadow layouts, and the fixed `AUDIT_EDITOR_OPEN_DELAY_MS` (450ms) guess at "has the smooth scroll settled" all need a real click-through before trusting the feel.
- De-duplication by `apiName + type` means only the FIRST on-page occurrence of a repeated entry is ever the navigation target — a deliberate simplification (documented, not a bug) worth knowing if the same field renders twice and only one occurrence is reachable by scroll.
- Audit scope is the CURRENT PAGE only, by design — reaching an off-page issue in Translation Health's org-wide list would need Salesforce's own navigation first, a materially different mechanism, not built here.

### #61 — Translation Audit real-org bug round: sticky-header scroll fix, editor-closes-on-click root cause (Dynamic Hover parity), Complete-tab overflow, session-local translation scope toggle (2026-07-21)
**Tags:** translation-mode, hover, tooltip-ui, dom-heuristics, product
**Files:** `content/index.tsx`, `content/dom-utils.ts`, `content/translation-mode.tsx`, `content/AuditPanel.tsx`, `content/audit-panel.css`

First real-org test of `#60` (Translation Audit), found via actual guided-navigation use on a real Lightning record page rather than the static mockups the prior entry relied on — exactly the kind of feedback loop the "real-org UNVERIFIED" caveats on `#59`/`#60` were flagging as needed. Four issues, all investigated for root cause per the session brief rather than patched superficially.

**1. Navigation into pinned/sticky headers didn't actually scroll the viewport.** Root cause, confirmed by re-reading `scrollIntoView`'s actual browser behavior rather than assumed: its built-in "walk every scrollable ancestor" algorithm can decide a `position: sticky`/`fixed` target is "already visible" purely from its own bounding-rect math, or resolve the wrong scrolling ancestor on a page with nested scroll containers (both apply to Salesforce Lightning: a pinned global header plus record pages that may scroll an inner container, not the window) — the highlight overlay looked correct because it's computed independently from `getBoundingClientRect()` on every scroll event regardless of what actually scrolled, which is exactly why "the target is visually surrounded but the viewport never moved" was possible: the highlight WAS tracking correctly, `scrollIntoView` just hadn't actually moved anything for that particular target.

Fix, in `content/index.tsx`: `scrollIntoView` stays the FIRST pass (cheap, works for the common case) but is no longer trusted alone — a verify-and-correct pass (`ensureVisibleAboveObstruction`) runs once it should have settled (`SCROLL_SETTLE_CHECK_MS`), measuring where the target ACTUALLY ended up:
- `measureTopObstruction()` samples a few x-offsets near the top of the viewport via `deepElementFromPoint` (the SAME shadow-piercing hit-testing primitive the hover engine itself already uses — reused, not duplicated) and walks each hit element's ancestor chain for `position: fixed`/`sticky`, taking the tallest such element's bottom edge as "how much pinned chrome currently covers the top." Deliberately generic (no hardcoded Salesforce class names, which could break across orgs/themes/releases) — it detects ANY pinned chrome, whatever produced it.
- `findScrollableAncestor()` (new export, `dom-utils.ts`'s `parentAcrossShadow` made public rather than reimplemented — the exact "prefer reusing shared logic" instruction from the brief) walks up from the target checking real `overflow-y`/`scrollHeight` to find the ACTUAL scrolling container, falling back to `document.scrollingElement`. Never assumes the window is what needs to scroll.
- If the target's rect isn't clear of the measured obstruction (or the bottom edge), `ensureVisibleAboveObstruction` applies one direct corrective `scrollBy` on that real ancestor. Symmetric by construction — it doesn't know or care which direction the navigation came from, only where the target ended up, which is what makes "Header → body" and "Body → header" behave identically instead of needing separate-cased logic.
- The editor-open timer now waits for this correction (an extra `SCROLL_CORRECTION_SETTLE_MS` only when a correction actually fired) before computing its anchor rect, so it never anchors to a pre-correction position.

**2. The Complete tab overflowed past the panel's right edge, misaligned with Next.** Root cause: `.sti-audit-panel__tabs` was `display: flex` with no wrap/shrink constraint — four pill buttons with count badges can easily sum past a 300px panel's content width once counts hit double digits, and flex's default behavior lets a row overflow rather than constrain it. Fix: `display: grid; grid-template-columns: repeat(4, 1fr)` — a fixed number of EQUAL-width columns whose total is ALWAYS exactly the container's content width, by construction, matching the stepper row below it (same padding). Also restructured each tab from "Label NN" inline to a stacked label-then-count layout (`sti-audit-tab__label`/`sti-audit-tab__count` on separate lines) — inline text was still truncating awkwardly with realistic double-digit counts even once the row itself stopped overflowing, since a quarter-panel column is narrow; stacked, each column only needs to fit the wider of the two short lines alone. Neither fix is a pixel-width patch — both are structural (proportional grid + a layout that doesn't need to fit both pieces of information on one line).

**3. Root cause of "clicking inside the modal closes it": `reconcileAfterEdit()`'s guard was wrong for the TM/audit editor, not a click-outside/shadow-DOM bug.** Investigated Dynamic Hover's actual ownership model first, per the brief's explicit instruction, before touching anything: the tooltip already persists through ANY inside click by construction (`#54` — closed shadow root event retargeting means `e.target === shadowHost` for the document-level outside-click listener, regardless of what was clicked inside it; this was NEVER the mechanism that was broken). The REAL bug: `TranslationEditor`'s textarea has `onBlur={commit}`, which calls `onCancel()` when the value is unchanged — losing focus for ANY reason, including clicking a completely different, harmless control inside the SAME tooltip (a different row's Copy button, for instance), sets `editingLang` to `null`. That triggers `onEditingActiveChange(false)` → `reconcileAfterEdit()`, which — before this fix — checked ONLY `isEngineLive()`. `isEngineLive()` is **always false** while Translation Mode is on, by design (TM suppresses the hover engine entirely) — so finishing ANY edit while the TM/audit editor was open unconditionally tore the WHOLE tooltip down via `clearTooltip()`, regardless of whether `tmEditorOpen` said it should still be showing. Dynamic Hover never hits this because there, `isEngineLive()` stays `true` while Inspection Mode is toggled on, so the exact same reconcile call correctly falls through to "keep it open."

Fix: `reconcileAfterEdit` now checks `!isEngineLive() && !tmEditorOpen` — a one-line, root-cause fix, not a new interaction model bolted on beside the existing one. This IS how Translate All's editor reaches genuine behavioral parity with Dynamic Hover: both now tear the tooltip down only when NEITHER surface that could legitimately still own it wants it kept.

Also added, since navigating away is exactly the kind of explicit action the brief says SHOULD close the current edit (matching `#55`'s Escape/outside-click precedent, not a new rule): `handleAuditNavigate`/`handleAuditFilterChange` now call `requestCancelEdit()` first when `isEditingActive` — previously, `openTmEditor`'s own `isEditingActive` guard meant clicking Next/Previous/a different filter while mid-edit silently refused to open the new target's editor at all, leaving the OLD one open pointed at whatever the highlight/scroll had already left behind.

**4. Translation scope (all fields vs. current field) — evaluated, then implemented as a session-local UI toggle, NOT a persisted `Settings` field.** The two-mode idea holds up: "All fields" (the existing, sensible default) is a broad-audit view; "Current only" declutters the PAGE's own inline badges down to just the audit panel's current target, valuable specifically while stepping through the guided stepper on a busy record page. Where it landed differently from the brief's own example (a `Settings`-style radio pair): this is a LIVE workflow control the user flips while actively using Translate All in the moment, not a stable cross-session preference — closer in spirit to `auditFilter`/`auditExpanded` (already session-local, unpersisted state for the exact same reason) than to `Settings.tmPreset` or the popup's other toggles. A single click-to-flip pill button in the panel's own header (`sti-audit-scope-btn`) satisfies "switching modes is easy" and "avoid a complex configuration panel" more directly than a popup radio group would — no settings trip required for something you'd want to change while looking at the page.

Architecture, chosen specifically to avoid re-scanning/re-fetching on every navigation or scope flip: `translation-mode.tsx` gained `setBadgeScope(scope, focusedElement)`, a PURELY presentational `display: none` toggle over the ALREADY-BUILT badges in `injectedBadges` — scope changes and navigation are instant, local operations with no network round trip. Re-applied at the end of every `scan()` (a rescan rebuilds badges as visible by default) so a save-triggered rescan mid-"current only" session can't silently undo it. The audit panel's own list/counts are computed from `auditEntries` directly and are UNAFFECTED by scope either way — scope only ever changes what's visually injected onto Salesforce's own DOM, never what the audit itself knows about or reports.

**Known limitations, real-org UNVERIFIED (same caveat as every round, but items 1 and 3 specifically need a confirmed real click-through since they were found BY real-org testing and fixed from code-reading + reasoning, not from watching the fix work live):**
- The sticky-header scroll fix's obstruction-measurement heuristic (sampling a few x-offsets, checking `position: fixed`/`sticky` on the hit ancestor chain) is generic by design but unverified against the FULL range of Salesforce header/path-assistant/utility-bar configurations — confirm it against pages with a compact sticky highlights panel specifically, and against a related list or other component with its own internal nested scroll.
- The reconcileAfterEdit fix was root-caused via careful code tracing (the exact blur → commit → cancel → reconcile chain), not reproduced live — confirm clicking a different row's control while a textarea is focused genuinely keeps the tooltip open now, including Save/Cancel/Copy buttons on OTHER rows, not just the one being edited.
- The scope toggle deliberately does NOT cancel an in-progress edit — unlike filter changes/navigation, flipping scope touches no highlight/scroll/index state at all, only which badges are visible, so the open editor stays valid and pointed at the same entry either way. Worth a specific real-org check that this reasoning holds (e.g. that the currently-edited entry's own badge doesn't itself get hidden by a "current only" toggle in a way that reads as inconsistent).

Second real-org feedback round on Translation Audit (`#60`/`#61`), this time explicitly about making it feel *inevitable* rather than merely correct. Each change below replaces a per-case behavior with ONE rule.

**1. Modal ownership is a MOUSEDOWN rule, not a click-outside rule.** Reported symptom: "the modal stays open when I click the translation input, but clicking other areas of the modal closes it." `#61` had already ruled out the click-outside mechanism (`#54`'s `e.target === shadowHost` identity check is exact and was never broken), which leaves focus, not clicks: `TranslationEditor`'s `onBlur={commit}` fires on ANY focus loss, and `commit()` with an unchanged value calls `onCancel()` — which closes the editor. So every mousedown inside the tooltip that wasn't aimed at the textarea itself (metadata text, the type badge, a Copy button, another row's pencil) tore the editor down as a side effect of a blur the user never asked for. `#61`'s `!tmEditorOpen` fix kept the tooltip SHELL alive through this, which is exactly why the remaining symptom read as "sometimes it survives, sometimes it doesn't."

Fix, in `Tooltip.tsx`: ONE `onMouseDown` on the tooltip ROOT that calls `preventDefault()` for any target that isn't a real form field, while an edit is open. Swallowing mousedown's default action is what prevents the focus change in the first place — the blur never happens, so nothing downstream of it can fire. Deliberately structural: the per-element `onMouseDown` guards Save/Cancel already carried WERE the failure mode in miniature (they only ever protected the controls someone remembered to annotate), so they were removed in favor of the root handler rather than copied onto every other element. Scoped to while-editing only, so view mode stays a normal surface where a translation value can still be selected with the mouse.

This also simplified the outside-click rule: now that an inside click provably cannot end an edit, an outside click is unambiguous, so it closes the whole surface in ONE gesture instead of `#55`'s two-stage cancel-then-close. Escape stays laddered (cancel edit → close inspector → clear search → collapse panel): a keystroke should unwind one level at a time, a click on genuinely different page content should not need repeating.

**2. Modal lifecycle = state transitions, not scattered conditions.** Each panel action was doing its own partial cleanup: filter change cancelled the edit but left the modal open; scope change and panel collapse did nothing at all; navigating to a `complete` or non-editable entry scrolled away and left the PREVIOUS entry's modal floating over the new target. The root shape of the bug is that "what invalidates the inspector" was encoded once per handler instead of once, period. Now one `invalidateAuditContext()` in `content/index.tsx` — bump the navigation token, cancel pending timers, drop the highlight, force-close the inspector — is called by every real transition (navigate, filter, search, scope, collapse, Translation Mode off) and by nothing else. Interactions INSIDE the inspector never call it; that boundary IS the rule.

`forceCloseInspector()` deliberately outranks the `isEditingActive` ownership guard that otherwise protects the tooltip from incidental clears: an explicit transition means the user asked for something that makes this editor meaningless. Safe because the unmount fires `CandidateBlock`'s own `onEditingActiveChange(false)` cleanup immediately after, so no flag is left stuck.

**3. Guided navigation is an abandonable async chain (navigation token).** `scrollIntoView`'s smooth behavior still has no completion callback, so navigating is inherently scroll → wait → verify → maybe correct → wait → open. `clearTimeout` alone can't express "abandon a chain that is already several steps in," so each step re-checks a monotonically-bumped `auditNavToken` and bails if the world moved on. This removes the stale-modal-from-a-previous-target class of bug at the source instead of guarding its symptoms one at a time.

**4. The panel's internal list follows navigation, the same way the page does.** `AuditPanel` keeps its own active row visible via `scrollRowIntoList`, keyed on the active entry's IDENTITY, not its index — a save reshuffles the filtered array, and scrolling to whatever now sits at the same index would be wrong. Deliberately NOT `scrollIntoView`: that walks up and scrolls every scrollable ancestor it finds, which here means the Salesforce page itself moving as a side effect of a panel-internal adjustment, directly fighting the obstruction-corrected page scroll guided navigation just computed. So it adjusts the list's own `scrollTop` and nothing else, and no-ops when the row is already comfortably visible (no animation for its own sake).

**5. Integrated search as a second AXIS, not an alternative to the filters.** One always-visible box between the header and the tabs — search first, then category, so the visual order matches the logical order. It matches the API name, the DISPLAY form of the API name (which drops the object prefix, so `Account.Phone` must still match a row rendered as just "Phone"), the type label, and every translated value: everything about an entry a user could plausibly type, with no configuration to get wrong. Filter counts are computed over the SEARCHED set, so switching tabs mid-search isn't a blind guess. The filter predicate now lives in ONE exported function (`filterAuditEntries`), used both by the panel's rendering and by `content/index.tsx`'s "what is the current target / which element gets the badge scope" — they previously each held their own copy of it, and adding a second axis to two copies is precisely how they drift apart.

**6. Status visual language: rails plus one active colour, not a coloured dashboard.** Rows lost their 6px status dot in favor of a continuous colour RAIL down the leading edge. A dot is a single point the eye has to find and then decode; a rail turns a scrolling list into visible BANDS of same-status rows, which is what "understand the page's translation state at a glance" actually requires. The same colour vocabulary carries to the current-target card (same rail) and to the ACTIVE filter tab only — four permanently-coloured tabs would be exactly the dashboard the brief warned against, while the selected one carrying its status colour says which lens you're looking through without reading the label. "All" stays neutral: it is the absence of a filter, not a status. Rows also gained a right-aligned language-code suffix (`ES FR`), turning "this row is missing something" into "this row is missing es, fr" with no second line and no hover. No filled status backgrounds anywhere; the ACTIVE row is marked by weight + tint + an inset border instead, so "which one am I on" never competes with "what state is it in."

**7. The native Translate All ↔ inspection question, answered by DELETING a condition rather than adding a feature.** The brief asked whether Translate All needs its own lightweight inspect action (a hover affordance, a per-row Inspect button, a configurable shortcut, a bridge to Inspect Core). It does not — and the friction that made it look necessary was a self-inflicted asymmetry: navigating to an editable `missing`/`identical` entry opened the full inspector (type, API name, every language's value, Setup link, edit), while navigating to a `complete` or non-editable one opened NOTHING and merely scrolled the page. One click, two meanings — and for the second kind, the only way to learn what you were looking at was to leave Translate All and turn on Dynamic Hover: exactly the mode-juggling the brief wanted eliminated. So `openAuditEditorIfApplicable` became `openAuditInspector`: navigation ALWAYS opens the same tooltip surface, and status/editability now decide only whether it lands pre-opened on the language that needs work or in read-only view mode. Zero new UI, zero new modes, zero new shortcuts, zero new state machines — Inspect Core already WAS the inspection system, it was just being conditionally withheld.

Rejected alternatives, for the record: a per-row "Inspect" button (a second click for what the first click should already do, plus a second meaning for one row); a dedicated inspect shortcut (a new global key and another mode for the user to manage, against PHASE 17's own conflict-avoidance direction); and letting Dynamic Hover run during Translation Mode (they would contend for the same tooltip — `isEngineLive()`'s TM suppression exists precisely to prevent that).

**Known limitation, deliberately not "fixed" harder.** `#61`'s sticky/pinned-header correction now runs a SECOND verify-and-correct pass, because a corrective scroll inside a nested container can itself bring the target under chrome that wasn't in the way when the first measurement was taken (Lightning's highlights panel collapses as the page scrolls, changing the obstruction height mid-flight). There is deliberately no third pass: beyond two, this is chasing an animation rather than converging. If a real org still lands a target under pinned chrome after two passes, the honest diagnosis is that Lightning's scroll-linked layout is still settling when we measure, and the correct fix would be an obstruction `MutationObserver`/`scrollend`-driven wait, not more retries. Recorded as a known limitation in `ROADMAP.md` PHASE 18 rather than patched speculatively.

**Real-org status:** unverified, like everything else in this codebase. Typecheck, unit tests and build pass, and every change above was root-caused by tracing the actual code path — but none of it has been clicked through against a live org.

### #63 — The closed-shadow-root duplicate-host bug (root cause of "clicking inside the modal closes it"), one interaction priority model, a real-browser dev harness, scroll-anchored modal + selection highlight, metadata-type filters, keyboard navigation (2026-07-21)
**Tags:** shadow-dom, hover, tooltip-ui, translation-mode, editing, testing, product
**Files:** `content/index.tsx`, `content/Tooltip.tsx`, `content/AuditPanel.tsx`, `content/interaction.ts` (new), `content/dom-utils.ts`, `content/audit-panel.css`, `content/interaction.test.ts` (new), `content/audit-panel.test.ts` (new), `dev-harness/` (new), `vite.harness.config.ts` (new)

Reported as "the Dynamic Hover modal used to survive clicks inside it and no longer does — the Translate All work broke it." It had, and the cause was not where any amount of reading the click logic would have found it.

**1. Root cause: `ensureShadowRoot()` was silently creating a SECOND host, because a closed shadow root cannot be read back off its host.**

```
if (shadowHost?.shadowRoot) return shadowHost.shadowRoot;   // ALWAYS null for mode:"closed"
```

`Element.shadowRoot` returns `null` for a root attached in closed mode — that is the entire point of closed mode, and it turned this cache check into "always a miss". Every caller therefore built its own host: `#sti-root` appeared TWICE in the DOM, each with its own closed root, and the module's `shadowHost` variable pointed at whichever was created last.

Click ownership is decided by `e.target === shadowHost` identity (`#54`) — clicks inside our UI retarget to their own host, which is exactly what makes that check exact and immune to stale geometry. With two hosts, clicks inside the surface whose host was NOT the last one created retargeted to a DIFFERENT element, failed the identity check, and were treated as clicks on the page: **dismiss the tooltip.**

That explains every confusing property of the report:
- It appeared only after Translate All (`#60`) shipped — before that there was one mount, so one host, so the bug had no way to express itself. The defect was older than `#60`; `#60` merely gave it a second caller.
- It affected Dynamic Hover, which nobody had touched: `applyTranslationModeSetting(true)` calls `clearTooltip()` (creating the tooltip's host) before `renderAuditPanel()` (creating the panel's), so once Translate All had been switched on in a page session, `shadowHost` pointed at the PANEL and every click inside the hover tooltip read as outside.
- It looked intermittent and area-dependent, because whether a given click closed the tooltip depended on which surface's host had been created last in that particular session.

The fix is to keep our own reference to the `ShadowRoot` returned by `attachShadow` instead of asking the host for it. The architecture doc's claim that both mounts share one root is now actually true rather than merely intended.

Two lessons worth carrying: a cache keyed on a value the platform deliberately hides will always miss; and identity-based ownership (`#54`) is only as sound as the guarantee that there is exactly ONE identity.

**2. A real-browser dev harness (`dev-harness/`, `npm run harness`).** The bug above cannot be found by reading, and cannot be reproduced in jsdom — jsdom never performs mousedown's focus default action, so the very mechanism under test is absent, and a test asserting "the editor survives an inside click" passes there whether or not the product works. So the harness runs the ACTUAL content script against a deliberately Lightning-shaped page (sticky global header, sticky highlights panel, tall body, nested scroll container) in a real browser, with `chrome.*` stubbed and fake metadata. It found, in one session: the duplicate-host bug, a navigation chain that opened no inspector at all, and `requestCancelEdit` silently doing nothing (see below). None of the three was visible in review; all three are one-line-ish fixes once seen.

It captures the closed root via `attachShadow`'s return value without weakening the mode, so assertions can reach inside a genuinely closed root — the same trick the product itself now uses, for the same reason.

Known limits, worth stating so future sessions don't over-trust it: the harness tab runs BACKGROUNDED (`visibilityState: "hidden"`), so the browser fires no scroll events, never runs `requestAnimationFrame`, and treats `behavior: "smooth"` as a no-op. Scroll-driven behaviour must be exercised by dispatching `scroll` events explicitly, and smooth-scroll ANIMATION cannot be observed there at all (the decision behind it is unit-tested instead — `listScrollDelta`). It is a development tool, imports nothing from the extension's build, and ships in no artifact.

**3. `requestCancelEdit` on the inspector surface never worked.** It relied on `host.render(null)` followed by `host.render(element)` to force a remount into view mode — but React 18 batches both calls into a single update and renders only the final element, so no remount happened, `autoEditLanguage` (read only in `useState` initializers) kept its old value, and the editor stayed open. Escape and outside-click therefore did nothing at all on that surface. Fixed by giving both surfaces the SAME cancel mechanism: the `cancelTrigger` counter the hover path already used, re-rendered onto the same mount. A separate, explicit `key` now expresses the cases that genuinely want a fresh mount (opening an editor, so clicking the same chip twice re-opens it) — `root.render()` reconciles its single child by key, which is a real remount guarantee rather than an assumed one.

**4. One interaction priority model (`content/interaction.ts`).** Ownership of clicks and keys is now decided by pure functions over a snapshot, not by sequential `if`s at each listener, with the hierarchy stated once in the file header: active editor → modal surface → the explicit selection → panel keyboard navigation → hover engine → the page. `content/index.tsx` builds the snapshot and dispatches. The rules are unit-tested (`interaction.test.ts`), which is what turns "we broke Dynamic Hover again" from a bug report into a failing test — and the outside-click regression from `#62` is pinned by a test asserting BOTH surfaces behave identically.

**5. Outside-click restored to two stages, reversing `#62`.** `#62` made one outside click destroy the whole surface even mid-edit, arguing the gesture was unambiguous. On the evidence it is unambiguous about the EDIT ending, not about the user being done with the metadata they were reading — the common motion is "click away to stop editing, keep reading the values", and `#62` silently threw the panel away instead. Back to `#55`: first click cancels the edit, second closes. Escape follows the same ladder, one level per press (cancel edit → close inspector → clear search → collapse panel).

**6. The modal follows its subject while scrolling; the selection highlight hides when its subject leaves the viewport.** Two questions, one answer, because they are two views of the same selection.

The modal used to be point-anchored — placed where the target was when it opened, then static — so scrolling detached it from the field it described. It is now ELEMENT-anchored when it was summoned by an explicit Translate All selection (`anchorEl`), re-deriving its origin from the anchor's live rect on scroll, and hiding (not unmounting — a half-typed edit and the conflict banner must survive) while the anchor is off screen. Two freeze conditions are the design, not edge cases: it never moves while an edit is open (a textarea that slides as you type is hostile, and the active editor outranks everything), and never while the pointer is over it (scrolling with the wheel while hovering the modal would otherwise pull it out from under the cursor). The hover tooltip deliberately does NOT get this: it describes wherever the cursor happened to be and has no persistent subject to stay attached to.

The highlight rectangle asked the opposite question — it tracked its element off the edge of the screen, where a rect clipped against the viewport reads as a rendering glitch. A SELECTED item is not a hover target: it keeps tracking while its element is on screen and simply stops being drawn when it isn't. Nothing about the selection changes; the panel's selected row is what states which item is current, which is exactly why the row's selected state was strengthened (below) rather than leaning on the rectangle. Both behaviours share ONE predicate, `isElementInViewport` in `dom-utils.ts`, so they can never disagree about whether the subject is visible.

**7. Metadata-type filters as chips, using `typeLabel()` — the same string the rows' own badges show.** That is what makes them self-explanatory: you filter by clicking the name already visible on the rows. It also settles the "should Field include Custom Field" question — they are separate, mutually exclusive categories, because the badges already split them that way and a "Field" chip that also matched rows badged "Custom Field" would contradict the row it selected; selecting both chips is the union, so "all fields" stays one click away without inventing an overlapping category. Only types actually present on the page get a chip. Quiet outline styling by default; a selected chip adopts its own type's badge colours, the only colour that row ever shows.

**8. Keyboard navigation, with the priority rules that matter being the ones that DON'T fire.** Up/Down and Left/Right are synonyms for prev/next (the list is one-dimensional; a second axis would be two names for one motion), Enter activates the current row, and everything routes through the same `handleAuditNavigate` the buttons and rows call — no parallel path, so scroll correction, highlighting and the inspector behave identically however the user got there. It yields entirely while an edit is open and while focus is in a real Salesforce field. The one deliberate exception: while focus is in the panel's OWN search box, Up/Down still navigate (type to narrow, then arrow into the results without reaching for the mouse) but Left/Right do NOT, because those move the caret in a field the user is typing in.

"Is focus in our search box" is read straight off the closed root's live `activeElement` rather than tracked with React focus handlers and a module flag. A flag can go stale — a re-render, a replaced node, a focus change that never fires its handler — and every one of those failures presents as the keyboard mysteriously doing the wrong thing. The DOM already knows, and we hold the only reference to that root. (`document.activeElement` cannot answer it: from outside, focus inside a closed root reads as the host.)

**9. Keyboard shortcut CONFIGURATION deliberately not built this session.** The request was explicit about compact/60% keyboards where arrows need a modifier, and that concern is real. But building a five-binding recorder UI now would mean designing conflict rules against three existing shortcuts before a single person has reported an actual collision — configuration ahead of evidence, which `#57` already had to simplify back down once. Arrows and Enter are also the one set no user has to learn. The existing conflict-checked recorder (`ShortcutToggleRow`, `#57`) makes this a small, well-understood addition whenever a real collision shows up; recorded in `ROADMAP.md` PHASE 17 with that trigger stated.

**10. The selected row is now genuinely different, not merely tinted.** It is the row the page is scrolled to, the highlight surrounds, the inspector is open on, and the keyboard is acting on — so it lifts off the list as a white card with a shadow, a solid brand-blue leading edge and a ▸ caret only it ever carries, while every other row stays flat. On the selected row that blue edge deliberately overrides the status rail: "this is where I am" outranks "what state is it in", which the detail line and the row's own language codes both still state.

**Real-org status:** still unverified against a live Salesforce org — but for the first time, the interaction model itself was verified end to end in a real browser via the harness: inside clicks on both surfaces (text, metadata, buttons, and across the OTHER mount), the two-stage outside click, the Escape ladder, guided navigation opening the inspector, scroll-anchored following, highlight hide/restore, search × type × status filtering, and every keyboard-ownership rule including the ones that must stay silent.

A freshness footer states what "current" means (the index's last refresh) instead of implying live org data. This is the owner's "Value A → Value B, changed since session started" requirement, grounded in data the extension already has.
4. **The page is the product, visually**: `workspace.css` reuses the audit panel's status-rail convention (edited blue `#0176d3` / pinned slate / changed amber `#b8860b`), the tooltip's `TYPE_COLORS` badges and language dots, `typeLabel`-style type chips (`#63`'s "filter by the name the rows' badges show"), status tabs with counts (All/Edited/Pinned/Changed), search across API names AND values (including pin snapshots), per-row Copy API name + **Open in Setup** (absolute links now possible: the background persists `lastOrgOrigin` on every `LOAD_LABELS`; rows without a resolvable entry simply don't offer the link), relative timestamps, the two-stage clear, and a first-person empty state that teaches the two capture paths.
